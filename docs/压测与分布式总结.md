## 一、压测（压力测试）

本项目建立了一套**完整的 4 层压力测试体系**，覆盖从底层依赖基准到全链路集成极限的所有层级。

### 1.1 测试架构（4 层金字塔）

| 层级 | 定位 | 对应工具/脚本 |
|------|------|-------------|
| **Layer 0** | 调试/诊断工具 | `debug-bot-reply.py`, `test-bots-ws.py` |
| **Layer 1** | 外部依赖基准测试 | `deepseek-concurrency-test.py`, `llm-benchmark.py` |
| **Layer 2** | 模块专项压测 | `bot-stress-test.py` |
| **Layer 3** | 系统集成压测 | `chatroom-stress-test.py`, `load-test.py` |
| **Layer 4** | 容量极限测试 | `max-qps-test.py`, `rag8-max-accounts-test.py` |

### 1.2 压测脚本详解（`test/` 目录）

#### max-qps-test.py（极限并发压测 v3，361 行）

4 个测试维度：
- **L1**：REST API 登录 QPS 测试（并发 20 快速 / 50 全量）
- **L2**：WebSocket 连接上限测试（梯级 10 → 5000）
- **L3**：消息吞吐测试（30 连接 × 20 条消息 = 600 条）
- **L4**：极限并发梯度测试（10 → 300，按 20 步进）

支持 `--quick` / `--full` 模式，使用原始 STOMP 协议通信。

#### load-test.py（综合负载压测 v2，663 行）

5 个维度：
- **L1**：REST API 登录 QPS + 延迟分布
- **L2**：WebSocket 连接并发上限
- **L3**：WebSocket 消息吞吐（msg/s）
- **L4**：Bot 并发回复
- **L5**：极限并发梯度（逐步提升直到错误率 > 20%）

支持 `--quick` / `--max` / `--skip-bot` 模式。内置 50 条多样化题库防缓存命中。

#### bot-stress-test.py（Bot 模块压测 v1，671 行）

5 个维度：
- **B1**：Bot 注册并发（同时注册 N 个 Bot）
- **B2**：Bot 消息并发（同时向 N 个 Bot 发送消息）
- **B3**：单 Bot 队列压测（快速发送 M 条消息，测试信号量和队列深度）
- **B4**：熔断器测试（错误 API Key 验证熔断行为）
- **B5**：极限并发梯度（逐级增加活跃 Bot 数量，直到回复率 < 70%）

内置 50 条多样化题库 `STRESS_MSG_POOL`，支持 `--quick` / `--blast` / `--skip-cb` 模式。

#### chatroom-stress-test.py（20-Bot 全链路聊天室压测 v3，约 65KB）

这是**最大的测试脚本**，核心设计要点：

- **20 种多样化 Bot 人设**：阳光开朗、温柔知心、冷幽默吐槽、毒舌怼人等
- **200+ 条不重复题库**：
  - `GENERIC_POOL`（40 条日常闲聊）
  - `TOPIC_POOLS`（200 条，10 个主题分类：hobby / food / tech / emotion / drama / game / sports / travel / work / creative）
  - `PERSONALITY_POOLS`（20 种个性 × 15 条专属问题）
  - `ACTIVE_OPENERS`（30 条开场白）
- **防缓存设计（v3 新增）**：个性匹配 + 轮次轮换 + 全局去重 → **0% 缓存命中**
- **运行模式**：被动模式 / 主动模式 / both 模式

#### test-bots-ws.py

WebSocket Bot 消息路由压力测试，测试连接、消息路由、错误率。

#### 集成测试脚本

- `test-bots.sh` / `test-bots.bat`：Bot 系统集成测试（注册、蒸馏、20-Bot 并发、错误隔离、熔断器、消息路由）
- `setup-deepseek-bots.sh` / `setup-deepseek-bots.bat`：DeepSeek Bot 一键部署脚本

### 1.3 后端基准测试服务

#### BotBenchmarkService.java（195 行）

内置 Bot 性能基准测试服务：
- `runBenchmark()`：发送 N 条消息，测量发送延迟、Bot 回复延迟、端到端响应时间
- `runQuickBenchmark()`：轻量级健康检查，测量消息保存延迟、Bot 技能获取延迟
- 报告 **p50 / p90 / p99 / mean / min / max** 全维度延迟指标
- 信号量控制并发：最多 10 并发

#### REST 端点（BotController.java）

- `POST /api/bots/{botUserId}/benchmark` — 全量基准测试
- `GET /api/bots/{botUserId}/benchmark/quick` — 快速基准测试

## 二、分布式

项目是一个**分布式软件系统课程项目**，系统设计以分布式演进为核心思路。

### 2.1 分布式服务编排（`docker-compose.yml`）

系统由 **5 个独立服务** 编排部署：

```
   mysql:9.3       redis:7        rabbitmq:management
       │               │                │
       └───────────────┼────────────────┘
                       │
                 chatroom-server     ←     chatroom-client
```

- **健康检查 + 依赖链**：server 等待 mysql / redis / rabbitmq 全部就绪后才启动，client 等待 server 就绪
- **数据持久化**：MySQL 和 Redis 挂载本地卷

### 2.2 分布式消息路由

#### 双模式 WebSocket（`WebSocketConfig.java`）

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| `simple` | 内置 STOMP Broker，单机直接路由 | 开发 / 单机部署 |
| `relay` | **RabbitMQ STOMP Relay**，消息经 RabbitMQ 跨实例路由 | 生产级分布式部署 |

通过环境变量 `CHATROOM_WEBSOCKET_MODE` 切换，无需改代码。

#### 消息流转路径（Relay 模式）

```
Client A ──WS──> Server 1 ──STOMP──> RabbitMQ ──STOMP──> Server 2 ──WS──> Client B
```

### 2.3 分布式系统核心模式

| 模式 | 实现位置 | 说明 |
|------|---------|------|
| **消息队列** | `BotMessageQueueService.java` | 基于 Redis List 的 FIFO 队列，每 Bot 最大 100 条，支持阻塞出队 `BLPOP` |
| **分布式锁** | `BotMessageQueueService.java` | Redis `SETNX` 实现，key 格式：`bot:lock:{botUserId}` |
| **熔断器** | `BotManager.java` | 5 次连续错误 → OPEN（15s 静默）→ HALF-OPEN → 探测成功 → CLOSED；状态持久化到 Redis |
| **信号量并发控制** | `BotManager.java` | `ConcurrentHashMap<Long, Semaphore>`，每 Bot 1 并发，保证消息串行处理 |
| **异步处理** | `BotManager.java` | 广泛使用 `CompletableFuture.runAsync()` 异步 DB 写入 |
| **降级回退** | 全局 | 所有 Redis 依赖均有 DB 回退路径；Redis 不可用时自动回退内存模式 |

#### 熔断器状态机

```
   CLOSED ──(5次连续错误)──> OPEN ──(15s后)──> HALF-OPEN ──(探测成功)──> CLOSED
     ↑                         │                      │
     └─────────────────────────┘              (探测失败)──> OPEN
```

#### 消息队列结构（Redis）

```
Key: bot:queue:{botUserId}     Type: List (FIFO, max 100)
Key: bot:lock:{botUserId}      Type: String (SETNX distributed lock)
```

- `BotMessageQueueService.java` 基于 Spring `@ConditionalOnBean(RedisTemplate.class)` — Redis 不可用时自动退回到 `ConcurrentLinkedQueue` 内存队列

### 2.4 多级记忆系统（分布式存储后端）

来自 `docs/memory-system-design.md`：

| 层级 | 名称 | 存储后端 | 用途 |
|------|------|---------|------|
| **Level 0** | 工作记忆 | Redis String（分布式缓存） | 当前对话上下文，TTL 自动过期 |
| **Level 1** | 短期记忆 | Redis List | 最近 N 轮对话，FIFO 滚动 |
| **Level 2** | 长期记忆 | MySQL + Redis 缓存 | 持久化重要记忆，带重要性评分 |
| **Level 3** | 语义记忆 | MySQL embeddings | 向量相似度检索，用于 RAG |

**降级策略**：所有层级均设计 Redis → MySQL 的回退路径和异常容错。

### 2.5 分布式演进路线

README 明确标注了系统的分布式演进方向：

> "系统采用前后端分离架构，支持 Docker 容器化部署，面向 **Redis / MySQL / RabbitMQ** 的分布式演进。"

`pom.xml` 中 RabbitMQ 依赖标注为 `<!-- RabbitMQ (optional, for production) -->`，表明这是一个可选的、面向生产级分布式扩展的组件。

## 三、压测与分布式的关系

| 压测验证的分布式能力 | 说明 |
|---------------------|------|
| **消息队列吞吐** | 验证 Redis List 队列在高并发下的入队/出队性能 |
| **分布式锁竞争** | 验证 SETNX 锁在 Bot 并发消息场景下的正确性 |
| **熔断器行为** | 验证断路器在分布式 LLM 调用失败时的状态转换 |
| **多服务协同** | `chatroom-stress-test.py` 验证 server + mysql + redis + deepseek 全链路 |
| **水平扩展能力** | WebSocket Relay 模式为多 server 实例的分布式部署提供基础 |
