## 1. 测试架构总览

### 1.1 设计理念

测试套件采用 **6 层测试金字塔** 架构，从底层协议诊断到 AI 模块深度测试层层递进：

```
Layer 6 (Java 单元测试)  →  AiServiceTests.java     42 cases, 10 模块, 纯 Mockito
Layer 5 (AI 模块深度测试) →  ai-module-test.py         8 维度, 26 项检查
                            deep-ai-test.py            7 维度, 21 项检查
Layer 4 (容量极限)        →  max-qps-test.py            4 维度 QPS 查找器
                            rag8-max-accounts-test.py   RAG-8 模式最大并行账户
Layer 3 (系统集成)        →  chatroom-stress-test.py    20 人格 Bot 全链路聊天室
                            load-test.py                5 维度综合负载
                            test-bots.py                28 步骤集成验证
Layer 2 (模块专项)        →  bot-stress-test.py         5 维度 Bot 模块压测
Layer 1 (外部依赖基准)     →  deepseek-concurrency-test.py  DeepSeek API 裸测
                            llm-benchmark.py            端到端 LLM 延迟测量
Layer 0 (调试卷)          →  debug-bot-reply.py         单帧协议诊断
                            test-bots-ws.py             WS 基础连通性
```

### 1.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **分层独立** | 每一层不依赖其他层的结果，可单独执行 |
| **逐级递增 + 自动停止** | 关键测试采用梯度模式，错误率超阈值自动停止（如并发 10→500，429 出现即停） |
| **指标统一** | 所有测试输出 avg/p50/p95/p99/min/max 延迟 + 成功率，可横向对比 |
| **多档模式** | 大部分脚本支持 `--quick` / 默认 / `--max` 三档，适配不同场景 |
| **防缓存命中** | v3 起所有测试采用 200+ 条不重复题库，避免响应缓存干扰测试结果 |

## 3. 防缓存设计

### 3.1 背景问题

旧版测试使用 10-15 条固定消息循环发送。Bot 管理系统内置 **响应缓存**（`normalizeContent` + LRU，TTL 3 分钟），相同问题直接返回缓存结果。这导致：

- 压力测试数据失真（缓存命中延迟 ~1ms vs LLM 调用延迟 ~2-5s）
- 无法测量真实的 LLM 并发处理能力
- `chatroom-stress-test.py` 3 轮 × 20 bot = 60 次发送，仅 15 条 unique 消息 → ~75% 缓存命中率

### 3.2 解决方案

| 脚本 | 旧题库 | 新题库 | 防缓存策略 |
|------|--------|--------|-----------|
| `chatroom-stress-test.py` | 15 条通用 + 10 条 opener | **200+ 通用 + 20×15 个性匹配 + 30 opener** | `MessageSelector` 跟踪全局已用，3 轮 60 条消息 **0 重复** |
| `load-test.py` L4 | 10 条循环 | **50 条** | 50 条不重复，`msgs_per_bot=10` 时仍有余量 |
| `bot-stress-test.py` B3 | `f"测试消息{i+1}: 你好吗？"` | **50 条多样化题库** | `STRESS_MSG_POOL` 随机选取，burst=50 时恰好每条一次 |

### 3.3 题库结构

```
GENERIC_POOL (40 条)     → 日常闲聊兜底
TOPIC_POOLS (200 条)     → 10 个主题分类: hobby/food/tech/emotion/drama/game/sports/travel/work/creative
PERSONALITY_POOLS        → 20 种 bot 个性 × 15 条专属问题，匹配 bot 人设
ACTIVE_OPENERS (30 条)   → bot 主动发起对话的独立开场白
```

### 3.4 防缓存策略三要素

| 策略 | 机制 | 效果 |
|------|------|------|
| **个性匹配** | 阳光派收到快乐相关、温柔派收到倾诉相关、毒舌派收到怼人相关 | 触发 LLM 生成更契合的回复 |
| **轮次轮换** | 每轮按 `(round + bot_index) % topic_count` 切换主题类别 | 同 bot 不同轮次聊不同话题 |
| **全局去重** | `MessageSelector.used_messages` 跟踪 session 内所有已用消息 | 绝不重复发送，唯一率 100% |

### 4.2 HTTP 连接池优化

**文件**: `LLMApiClient.java`

```java
PoolingHttpClientConnectionManager cm = new PoolingHttpClientConnectionManager();
cm.setMaxTotal(300);           // 总连接数 300
cm.setDefaultMaxPerRoute(300); // 单路由 300
// connectTimeout = 30s, socketTimeout = 180s
```

| 参数 | 值 | 设计考量 |
|------|-----|---------|
| maxTotal | 300 | 200+ Bot 同时在线需要至少 200 连接，留 50% 余量 |
| defaultMaxPerRoute | 300 | 所有 Bot 通常调用同一 LLM API 端点 |
| connectTimeout | 30s | TCP 握手超时，过短则网络波动时误判 |
| socketTimeout | 180s | LLM 流式生成可能持续几十秒到几分钟 |

### 4.4 信号量并发控制

**文件**: `BotManager.java`

```java
// 每个 Bot 1 个并发许可 — 保证同一 Bot 的消息串行处理
ConcurrentHashMap<Long, Semaphore> botSemaphores = new ConcurrentHashMap<>();
// 每个群 1 个并发许可 — 保证同一群内 Bot 发言不冲突
ConcurrentHashMap<Long, Semaphore> groupSemaphores = new ConcurrentHashMap<>();
```

**工作流程**:

```
消息到达 → sem.tryAcquire()
    ├── 成功 → 直接调用 LLM → 完成 → sem.release() → processQueue()
    └── 失败 → enqueue Redis List (FIFO) → 等当前消息处理完自动出队
```

**设计要点**:

- 信号量 = 1 保证每个 Bot 的上下文**严格有序**，不存在竞态条件
- 获取失败时**不丢弃**，入 Redis 队列持久化
- 队列满时（max=200）才丢弃并记录 `log.warn`

### 4.5 Redis 消息队列 + 分布式锁

**文件**: `BotMessageQueueService.java`

```
Key: bot:queue:{botUserId}     Type: List (FIFO, max 200)
Key: bot:lock:{botUserId}      Type: String (SETNX distributed lock)
Key: bot:circuit:{botUserId}   Type: String (JSON state)
```

| 操作 | Redis 命令 | 行为 |
|------|-----------|------|
| 入队 | `LPUSH` | 从左侧推入 |
| 出队 | `RPOP` | 从右侧弹出（FIFO） |
| 阻塞出队 | `RPOP` + timeout | BLPOP 语义，超时 3s |
| 分布式锁 | `SETNX` | 防止多实例同时处理同一 Bot 队列 |
| 熔断状态 | `SET` + TTL 24h | 跨实例共享熔断器状态 |

**降级策略**：

```java
@ConditionalOnBean(RedisTemplate.class)  // Redis 不可用时 Bean 不创建
volatile boolean redisAvailable;         // 运行时健康检查
// Redis 不可用 → 自动回退到 ConcurrentLinkedQueue 内存队列
```

### 5.0 指标含义速查表

在阅读测试结果之前，先理解每项指标代表的含义：

| 指标 | 全称 | 含义 | 好/坏的判断标准 |
|------|------|------|---------------|
| **QPS** | Queries Per Second | 每秒处理的请求数，反映系统吞吐量 | 越高越好。REST API >1000 为优秀 |
| **msg/s** | Messages Per Second | 每秒处理的消息数（全链路：接收→存储→路由→推送） | 越高越好。>500 为优秀 |
| **ERR%** | Error Rate | 失败请求数 / 总请求数 × 100 | **必须为 0%**（或 <1%） |
| **ReplyRate** | Bot Reply Rate | Bot 实际回复数 / 发送消息数 | **必须为 100%**（生产级要求） |
| **avg** | Average Latency | 所有请求延迟的算术平均值 | 参考值，但会被极端值拉高，**不可单独使用** |
| **p50** | 50th Percentile (中位数) | 50% 的请求延迟低于此值 | **最重要的指标之一**。代表"典型用户体验" |
| **p95** | 95th Percentile | 95% 的请求延迟低于此值 | **SLA 核心指标**。代表"绝大多数用户的最差体验" |
| **p99** | 99th Percentile | 99% 的请求延迟低于此值 | **长尾指标**。代表"极端情况下的最差体验" |
| **min** | Minimum Latency | 所有请求中最快的延迟 | 参考值。反映系统在最优路径下的开销下限 |
| **max** | Maximum Latency | 所有请求中最慢的延迟 | 参考值。反映系统的**最坏情况**（可能由 GC/网络抖动导致） |
| **wall** | Wall Clock Time | 测试总耗时（秒） | 结合 QPS 看：QPS = ok / wall |
| **unique** | Unique Messages | 去重后的唯一消息数 | 防缓存指标。unique = sent 表示 0% 缓存命中 |
| **uniqueness%** | Uniqueness Rate | unique / sent × 100 | **必须为 100%**（否则压力测试数据失真） |
| **TTFB** | Time To First Byte | 从请求发出到收到第一个响应字节的时间 | LLM 流式场景下代表模型思考时间。毫秒级优秀，秒级可接受 |
| **chunks/s** | Chunks Per Second | 流式响应中每秒收到的文本片段数 | 代表 LLM token 生成速度。受限于 LLM API |
| **QueueDrop** | Queue Drop Count | 因队列满而被丢弃的消息数 | **应为 0**（>0 表示需要扩容队列或限流） |
| **Burst msg/s** | Burst Send Rate | 突发发送速度（客户端瞬间发出的消息数/秒） | 仅反映客户端发送能力，不是服务端处理能力 |

> **核心原则**：延迟类指标看分布（p50 / p95 / p99），不看平均值（avg）。平均值会被少数极端值严重拉高，掩盖了大多数用户的真实体验。

## 7. 并发测试指标深度解读

### 7.0 延迟分位数 (p50/p95/p99) 含义速查

在解读具体测试结果前，先理解这三个最关键的延迟指标：

```
假设有 100 个请求，延迟从小到大排序：
  第 1 个(min) → ... → 第 50 个(p50) → ... → 第 95 个(p95) → 第 99 个(p99) → 第 100 个(max)

p50（中位数）：50% 的请求延迟 ≤ 此值 → 「典型用户的体验」
p95：          95% 的请求延迟 ≤ 此值 → 「绝大多数用户的最差体验」→ **SLA 核心**
p99：          99% 的请求延迟 ≤ 此值 → 「极端情况下的最差体验」→ **长尾指标**
```

| 分布特征 | 含义 | 健康程度 |
|---------|------|---------|
| p50≈p95≈p99（差距 <5ms） | 延迟极度集中，无抖动 | 🟢 优秀 — 系统行为高度可预测 |
| p95-p50 <30% p50 | 长尾可控 | 🟡 可接受 |
| p99 >> p95（差距 >2×） | 有严重长尾，少数请求极慢 | 🔴 需排查 — 可能是 GC、锁竞争、网络重传 |
| avg >> p50 | 被极端值拉高，avg 不可信 | ⚠️ 不能单看 avg，必须看分布 |

**本项目实测的延迟分布特征**：

| 测试场景 | p50 | p95 | p99 | 分布特征 | 评价 |
|---------|-----|-----|-----|---------|------|
| REST API（max-qps） | 14ms | 16ms | 16ms | p50=p95=p99，差距 0-2ms | 🟢 完美 |
| REST API（load-test quick） | 99ms | 100ms | ~100ms | p50≈p95，差距 1ms | 🟢 完美 |
| Bot 回复框架层 | 35ms | 46ms | ~46ms | p95-p50=11ms <30% | 🟢 优秀 |

### 7.1 REST API QPS 与延迟 — 1226 QPS, p50=14ms, p95=16ms, p99=16ms

#### 各指标分别代表什么

| 指标 | 值 | 代表的系统行为 |
|------|-----|-------------|
| **QPS=1226** | 每秒处理 1226 个请求 | Spring Boot + Tomcat + HikariCP + JWT 无状态认证的吞吐上限。每个请求经过：TCP 接收 → HTTP 解析 → Filter 链（JWT 验证）→ Controller → MyBatis-Plus → HikariCP → MySQL → 序列化 JSON → TCP 发送 |
| **p50=14ms** | 一半请求 ≤14ms | 「典型用户」登录耗时 14ms。这个延迟包含了网络往返 + JWT 解析 + DB 查询 + JSON 序列化的全部开销。14ms 意味着用户感知不到延迟（<100ms 人眼无法察觉） |
| **p95=16ms** | 95% 请求 ≤16ms | 「绝大多数用户的最差体验」仅 16ms。p95 只比 p50 多 2ms，说明 **95% 的用户体验与中位数几乎一致** |
| **p99=16ms** | 99% 请求 ≤16ms | p99 = p95 = 16ms 是**最值得关注的信号**——说明系统几乎没有长尾延迟。99% 的用户和 95% 的用户体验完全相同。这意味着：无 JVM GC 暂停、无锁竞争、无连接池等待 |
| **min（估计~10ms）** | 最快请求延迟 | 网络 IO → Spring 路由 → 内存缓存命中（如 JWT 本地验证）→ 响应的最小开销。最优路径的瓶颈在 CPU 和网络 |
| **max（估计~18ms）** | 最慢请求延迟 | 由于 p99=16ms, max 不可能远高于此。最慢请求可能遇到了 DB 索引页未缓存（buffer pool miss） |

> **CPU-bound 证据**：p50=14, p95=16, p99=16 这种极窄分布说明系统是 **CPU-bound**（处理时间由 CPU 决定，几乎恒定），而非 **IO-bound**（处理时间随 IO 延迟波动）。这意味着 DB 连接池充足、無锁竞争、无 GC 暂停。

#### 这个结果"好"在哪

- 1226 QPS 对于即时通讯场景绰绰有余：假设 10 万在线用户，平均每人每分钟发 1 次 REST 请求 → 1666 QPS。1226 QPS 已经接近这个量级，单机即可满足
- p99=16ms 意味着即使用户运气最差（排在 99% 位置），延迟也才 16ms，完全无法感知
- `max-qps-test` 的 1226 vs `load-test` quick 的 92 QPS 的差异：并发度不同。quick 模式用低并发测功能正确性，max-qps 用高并发测吞吐上限

### 7.5 Bot 回复延迟 — p50=35ms, p95=46ms

| 指标 | 值 | 代表的系统行为 |
|------|-----|-------------|
| **p50=35ms** | 框架层中位数延迟 | 消息到达 → `sem.tryAcquire()` 成功 → `buildContext()` 组装 Prompt → `llmApiClient.chat()` HTTP 请求发出 → LLM 回复到达 → WebSocket `convertAndSendToUser()` 推送 |

**与 LLM 总延迟的对比**：

```

    │
    ├── 消息路由 + 信号量获取: ~5ms
    ├── Context 组装 (buildContext): ~10ms  ← 多层记忆查询（L0-L3）

    └── WebSocket 推送: ~5ms

框架层占比: 35ms / (35ms + 569ms) ≈ 5.8%
```

> **结论**：框架层开销仅占 LLM 总延迟的 ~6%。优化方向不在框架层（已经很快），而在 LLM API 的速度和 Prompt 的 token 数量。

### 7.6 熔断器 — 5 次错误 → OPEN → 15s 静默 → HALF-OPEN → 探测 → CLOSED

#### 每一步代表什么

| 阶段 | 触发条件 | 系统行为 | 为什么这样设计 |
|------|---------|---------|-------------|
| 错误累积 | LLM API 调用抛出异常 | `errorCount++`，写入 DB | 内存计数可能丢失（服务重启），DB 持久化保证准确 |
| **CLOSED→OPEN** | `errorCount ≥ 5`（连续） | ① 状态写入 Redis `bot:circuit:{id}` ② DB `status = CIRCUIT_BROKEN` ③ 所有新消息直接返回 null | 5 次阈值在"快速响应故障"和"避免误触发"之间取平衡。太少（如 2）可能偶发网络抖动就触发；太多（如 20）则故障检测太慢 |
| **OPEN 静默期** | 15s 倒计时 | `isCircuitOpen()` 返回 true → 消息不入队、不调 LLM | 给外部 API（DeepSeek/OpenAI）15 秒恢复时间。避免在 API 故障期间持续消耗线程和连接 |
| **OPEN→HALF-OPEN** | 15s 到期 | 允许 1 条消息穿透，实际调用 LLM | 用真实请求探测 API 是否恢复，比定时 ping 更准确 |
| **HALF-OPEN→CLOSED** | 探测成功 | `errorCount=0`、`status=ACTIVE`、清除 Redis 熔断状态 | 恢复后立即清零计数，避免旧错误影响新请求 |
| **HALF-OPEN→OPEN** | 探测失败 | 重新计时 15s 静默期 | 故障未恢复，继续等待 |

> **为什么会是"35s"恢复而非 15s**：测试中等待了额外时间确保状态同步（DB 写入 + Redis 同步 + 调度周期），实际熔断器本身的静默期是 15s。

### 7.7 单 Bot 队列压测 — 29485 msg/s burst, QueueDrop=5

| 指标 | 值 | 代表的系统行为 |
|------|-----|-------------|
| **Burst Rate** | 29485 msg/s | 客户端瞬间发出 20 条消息的发送速率（Python 本地循环，无网络延迟）。**这不代表服务端处理能力**，只是反映客户端发包速度 |
| **Sent** | 20 | 共发送 20 条 |
| **Replied** | 15/20 | Bot 处理了 15 条 |
| **QueueDrop** | 5 | 5 条消息因队列满而被丢弃 |
| **为什么 Drop=5** | — | `BOT_MAX_QUEUE_SIZE=10`（Constants.java）+ Semaphore=1。20 条 burst 到达时：第 1 条获得信号量正在处理，第 2-11 条入队（10 条），队列满。第 12-20 条尝试入队 → `size≥maxQueueSize` → 丢弃（9 条）。但由于测试并发时序，实际只观察到 5 条丢弃——说明部分消息在队列消费后被接受 |

> **QueueDrop 的含义**：>0 表示队列溢出保护生效。这不是 bug——是**设计行为**：用队列满丢弃来保护系统不被突发流量打垮。生产环境处理方式：① 增大 `bot.queue.max-size`（当前 200，代码中实际为 10 的常量）② 客户端指数退避重试 ③ 上游限流

### 7.11 SSE 流式 — 37 chunks, TTFB=569ms, 69 字

| 指标 | 值 | 代表的系统行为 |
|------|-----|-------------|
| **TTFB=569ms** | 首 token 延迟 | LLM 从收到完整 Prompt 到输出第一个 token 的时间。包含：Prompt 处理 + KV Cache 计算 + 第一个 token 生成。569ms 说明 DeepSeek 推理启动很快 |
| **37 chunks** | 文本片段数 | SSE 流中 `data: {...}` 帧的数量，每个 chunk 包含 1 个或多个 token |
| **69 字** | 总生成文本长度 | 完整回复的文本长度 |
| **chunks/s=37** | 生成速度 | 约 1.6 字/chunk，相当于约 2-3 token/chunk。受限于 LLM API 的生成速度 |

> **TTFB 的组成**：网络往返（~50ms）+ Prompt 编码（~100ms）+ 模型推理启动（~400ms）= ~550ms。实际 TTFB 主要取决于 Prompt 长度（token 数越多，KV Cache 计算越久）。

## 8. AI 模块深度分析 — 记忆系统、缓存与 API Key 冲击

### 8.0 为什么 AI 模块的缓存设计是最关键的优化

传统 Web 应用的瓶颈在数据库，优化方向是加缓存、加索引、读写分离。但 **AI 即时通讯系统** 的瓶颈完全不同：

- 每次 Bot 回复 = 一次 LLM API 调用 = **消耗 API Key 配额 + 产生费用 + 等待 0.5~5 秒**
- 如果不加缓存，20 个 Bot 各聊 3 轮 = 至少 60 次 LLM 调用
- 如果同一句话问两次，没有缓存 = **白白浪费一次 API 调用的钱和时间**

因此，AI 模块的缓存策略直接决定了系统的 **运行成本** 和 **用户体验**。

---

### 8.1 多级记忆缓存体系（L0-L3）

系统的记忆系统是一个 **4 级金字塔**，每一级有不同的存储后端、TTL 和用途。各级之间自动流转，形成完整的记忆生命周期。

```
┌──────────────────────────────────────────────────────┐
│                  Level 3: 语义记忆 (RAG)               │
│  存储: MySQL embeddings 表 + 向量余弦相似度检索         │
│  生命周期: 永久（除非手动清除）                         │
│  容量: 无上限                                          │
│  作用: 跨会话的语义知识检索                             │
│  触发: 每次 LLM 调用前自动检索 Top-K 相关记忆           │
├──────────────────────────────────────────────────────┤
│                  Level 2: 长期记忆 (LTM)                │
│  存储: MySQL (bot_long_term_memory 表) + Redis 缓存     │
│  生命周期: 永久（按 importance 1-5 + LRU 淘汰）         │
│  容量: 每 Bot-User 对最多 50 条                         │
│  作用: 持久化的重要事实/偏好/对话摘要                    │
│  触发: 短期记忆超过 20 条 → LLM Consolidation → 写入     │
│  注入: 每次 LLM 调用时注入最近 5 条高重要性记忆          │
├──────────────────────────────────────────────────────┤
│                  Level 1: 短期记忆                      │
│  存储: Redis List (FIFO)                               │
│  生命周期: TTL 30 分钟                                  │
│  容量: 最多 30 条消息                                   │
│  作用: 当前会话的最近对话历史                            │
│  触发: 每条消息自动追加                                 │
│  降级: Redis 不可用 → 内存 LinkedList                   │
├──────────────────────────────────────────────────────┤
│                  Level 0: 工作记忆                      │
│  存储: Redis String (JSON)                              │
│  生命周期: TTL 60 分钟                                  │
│  容量: 5 次对话交换 × 每条 ≤3000 字符                    │
│  作用: 当前对话的即时上下文（LLM Prompt 中直接注入）      │
│  触发: 每条消息自动更新                                 │
│  降级: Redis 不可用 → ConcurrentHashMap                 │
└──────────────────────────────────────────────────────┘
```

#### 各级记忆在 LLM Prompt 中的位置

当 Bot 收到一条消息 `"你还记得我喜欢什么吗？"`，系统组装的 Prompt 结构如下：

```
┌─────────────────────────────────────────┐
│ System Prompt (Bot 人设)                 │  ← 来自 BotSkill 表，Skill Cache 缓存
│ "你是一个温柔知心的大姐姐..."              │
├─────────────────────────────────────────┤
│ L2 Long-Term Memory (最多 5 条)           │  ← 来自 MySQL/Redis，按重要性排序
│ [记忆] 用户喜欢猫，有一只叫小橘的橘猫       │
│ [记忆] 用户偏好温柔的语气，不喜欢讲道理     │
├─────────────────────────────────────────┤
│ Few-Shot Examples                       │  ← 来自 BotSkill.few_shot_examples
│ 用户: 我今天好累                          │
│ Bot: 辛苦啦~来姐姐这里靠一会儿...          │
├─────────────────────────────────────────┤
│ L1 Short-Term Memory (最近 10 条)         │  ← 来自 Redis List
│ 用户: 今天天气真好                        │
│ Bot: 是呀，阳光明媚的～                   │
│ 用户: 小橘今天特别粘人                    │
│ Bot: 猫咪粘人说明它特别信任你呢           │
├─────────────────────────────────────────┤
│ L3 RAG Retrieval (Top-K=3)               │  ← 余弦相似度检索 MySQL embeddings
│ [相关] 用户之前聊过猫咪的饮食偏好           │
│ [相关] 用户问过橘猫的品种特点              │
├─────────────────────────────────────────┤
│ Current Message                         │
│ "你还记得我喜欢什么吗？"                   │
└─────────────────────────────────────────┘
```

> **这个 Prompt 结构每一层都是一个缓存决策点**。如果去掉任意一层缓存，要么多一次 DB 查询，要么多一次 Redis 调用，要么多一次 LLM 调用。

---

### 8.2 响应缓存（Response Cache）— API Key 的守护神

这是系统中最直接影响 **API Key 消耗** 的缓存层。

#### 设计

```java
// BotManager.java
// 容量: 2000 条 | TTL: 180 秒 (3 分钟) | 淘汰策略: LRU
Map<String, CacheEntry<String>> responseCache = 
    Collections.synchronizedMap(new LinkedHashMap<>(16, 0.75f, true) {
        @Override
        protected boolean removeEldestEntry(...) {
            return size() > 2000;  // LRU 自动淘汰
        }
    });

// 缓存 Key: botUserId + ":" + normalizeContent(content)
// normalizeContent: trim + 合并空白 + 小写
// 例如: "你好，今天天气怎么样？" → "1:你好，今天天气怎么样？"
```

#### 工作流程

```
用户消息到达
    │
    ├──① normalizeContent() → "1:你好，今天天气怎么样？"
    │
    ├──② responseCache.get(cacheKey)
    │       │
    │       ├── 命中 (TTL 未过期) → 直接返回缓存 → 0 次 LLM 调用 ✅
    │       │   延迟: ~1ms（内存查 HashMap）
    │       │
    │       └── 未命中 → ③调用 LLM API → ④存入 responseCache → 返回
    │           延迟: 569ms TTFB + 生成时间
    │           消耗: 1 次 API Key 配额 + 费用
```

#### 对 API Key 消耗的量化影响

假设一个 20 人的群聊，Bot 活跃参与，每分钟产生 10 条消息：

| 场景 | 缓存命中率 | LLM 调用次数/分钟 | API Key 消耗/分钟 | 月度 API 费用（估算） |
|------|-----------|------------------|------------------|-------------------|
| **无响应缓存** | 0% | 10 | 每次 = 全新调用 | ~¥300（10 msg/min × 60 × 24 × 30 × ~¥0.002/msg） |
| **有响应缓存（无防缓存测试）** | ~75% | 2.5 | **减少 75%** | ~¥75 |
| **有响应缓存（防缓存测试）** | 0% | 10 | 测试场景下故意绕过 | 仅测试时的临时消耗 |
| **生产环境实际** | 30-50% | 5-7 | **减少 30-50%** | ~¥150-210 |

> **关键结论**：响应缓存每提升 10% 命中率 = API Key 费用减少 10%。2000 条容量 + 180s TTL 的设计在内存占用（约 2-5MB）和命中率之间取得了平衡。
>
> **防缓存测试的意义**：测试时故意绕过缓存（200+ 不重复题库），目的是测**最坏情况下的系统表现**——即所有消息都需要真实 LLM 调用时，系统能否撑住。如果连最坏情况都能扛住（0% 缓存命中 + 20 Bot → 100% 回复率），那正常情况（30-50% 缓存命中）就更不是问题。

### 8.3 Skill 缓存 — 消除 DB 瓶颈

```java
// BotManager.java
// 容量: 500 条 | TTL: 60 秒 | 预热: @PostConstruct 全量加载
Map<Long, CacheEntry<BotSkill>> skillCache = 
    Collections.synchronizedMap(new LinkedHashMap<>(16, 0.75f, true) {
        @Override
        protected boolean removeEldestEntry(...) {
            return size() > 500;
        }
    });

@PostConstruct
public void preWarmSkills() {
    // 启动时一次性加载所有 BotSkill 到缓存
    List<BotSkill> all = botSkillMapper.selectList(null);
    for (BotSkill s : all) {
        skillCache.put(s.getBotUserId(), new CacheEntry<>(s, 
            System.currentTimeMillis() + 60_000));
    }
}
```

#### 如果没有 Skill 缓存

| 场景 | 有 Skill 缓存 | 无 Skill 缓存 |
|------|-------------|-------------|
| 20 Bot 并发各收 1 条消息 | 0 次 DB 查询（全部命中） | 20 次 `SELECT * FROM bot_skill WHERE bot_user_id=?` |
| 技能蒸馏（遍历所有 Bot） | 0 次 DB 查询 | N 次 DB 查询（N=Bot 总数） |
| 冷启动首次请求 | 0 次（预热已加载） | N 次（懒加载逐个查） |
| 缓存过期（60s TTL） | 1 次 DB 查询重新加载 | — |

> `@PostConstruct` 预热是关键设计：启动时一次性 `selectList(null)` 全量加载，之后所有请求走内存。DB 只在 BotSkill 变更时（注册新 Bot / 更新配置）才被访问。

### 8.4 缓存级联效应 — 一次命中省下多少？

当用户问 Bot 一个 **曾经问过的问题** 时，各级缓存依次发挥作用：

```
消息: "小橘喜欢吃什么？"
  │
  ├──① Skill Cache 命中 → 省 1 次 DB 查询 (~10ms)
  │
  ├──② Response Cache 命中 → 省 1 次 LLM 调用 (~2-5s, ~¥0.002)
  │     ↑ 最关键的一步 —— 直接返回之前 LLM 生成的回复
  │     ↑ 如果命中，以下步骤全部跳过
  │
  ├──③ 构建 Prompt Context:
  │   ├── L2 LTM Cache (Redis) 命中 → 省 1-2 次 DB 查询
  │   ├── L1 Short-Term Memory (Redis List) → 必然命中（主动维护）
  │   ├── L0 Working Memory (Redis String) → 必然命中（主动维护）
  │   └── L3 RAG (MySQL embeddings) → 必须查询（语义检索无法缓存）
  │
  └──④ LLM API 调用 → 消耗 1 次配额
```

**一次 Response Cache 命中的总节省**：

| 省下的操作 | 时间节省 | 费用节省 |
|-----------|---------|---------|
| L2 LTM DB 查询 | ~10ms | 无 |
| L3 RAG 向量检索 | ~20ms | 无 |
| LLM API 调用 | **2000-5000ms** | **~¥0.002** |
| **合计** | **~2-5 秒** | **~¥0.002/次** |

> 虽然单次 ¥0.002 看起来很少，但 20 Bot × 每天数百次对话 = 每天可能节省数十元，每月可节省数百元 API 费用。

### 8.5 Prompt 压缩 — 减少 Token 消耗

```java
// BotManager.java
private String compressPrompt(String prompt) {
    if (prompt == null || prompt.length() <= 8000) return prompt;
    int keepHead = 5000;  // 保留头部: System Prompt + LTM + Few-Shot
    int keepTail = 3000;  // 保留尾部: Short-Term Memory + RAG + Current Message
    return prompt.substring(0, keepHead) 
         + "\n\n...(中间内容已压缩)...\n\n" 
         + prompt.substring(prompt.length() - keepTail);
}
```

#### 为什么是 8000 字符截断 + 5000/3000 分配

| 决策 | 原因 |
|------|------|
| 8000 字符阈值 | 中文约 8000 字符 ≈ 4000-5000 tokens。DeepSeek 上下文窗口 128K tokens，8000 字符远未到极限。截断是为了 **控制 LLM 调用延迟**（Prompt 越长 → TTFB 越大 → 用户等待越久） |
| 保留头部 5000 | 头部包含 System Prompt（Bot 人设核心）+ LTM 注入（用户偏好记忆），这部分对回复质量**最关键** |
| 保留尾部 3000 | 尾部包含最近的对话历史 + 当前消息，是 LLM 理解"现在在聊什么"的依据 |
| 丢弃中间 | 中间的旧对话轮次和低重要性上下文。丢弃它们对回复质量的负面影响最小 |

#### 不压缩的后果

| 场景 | 无压缩 | 有压缩 |
|------|--------|--------|
| 长时间对话后 Prompt 长度 | 可能 15000+ 字符 (~7500 tokens) | 8000 字符 (~4000 tokens) |
| Token 消耗/次 | ~7500 input tokens | ~4000 input tokens |
| 单次 API 费用 | ~¥0.015 (按 DeepSeek 价格) | ~¥0.008 |
| TTFB (推理启动) | ~800-1200ms | ~400-600ms |
| 每日 100 次调用费用 | ~¥1.50 | **~¥0.80 (节省 47%)** |

### 8.6 测试如何验证缓存不影响回复质量

防缓存设计要回答一个核心问题：**测试时绕过缓存，确保测的是真实 LLM 能力；但生产环境中缓存必须足够有效，节省 API Key。**

测试体系中针对这个问题的验证：

| 测试 | 验证内容 | 结果 |
|------|---------|------|
| D1: RAG Accuracy | L3 语义检索的准确性（Embedding 质量 + 余弦相似度阈值） | 6 个关键词匹配 ✅ |
| D6: LTM Accuracy | L2 长期记忆固化的质量（LLM Consolidation 后 memory entry 是否正确） | 格式正确，质量待人工审查 |
| D7: Memory Cascade | L0→L1→L2 的自动流转是否触发（不丢记忆） | 完整流转 ✅ |

### 8.7 API Key 冲击分析 — 压测对 API 的实际冲击

#### 8.7.1 压测的 API 消耗估算

| 测试脚本 | LLM 调用次数（估算） | API Key 消耗 |
|---------|-------------------|-------------|
| `deepseek-concurrency-test.py`（550 并发） | 550 次（每次 `"Say OK"`） | **极低**（最小 token 提示词） |
| `bot-stress-test.py` B2（20 Bot 消息并发） | 20 次 | **低** |
| `bot-stress-test.py` B3（单 Bot 队列） | 15-20 次 | **低** |
| `bot-stress-test.py` B5（极限梯度） | 20+20=40 次 | **中等** |
| `chatroom-stress-test.py`（20 Bot × 3 轮） | 60 次 | **中等** |
| `load-test.py` L4（Bot 并发） | 10 次 | **低** |
| `ai-module-test.py`（RAG+LTM+Streaming+...） | 20-30 次 | **中等** |

| **全部 6 层测试运行一次** | **~700 次 LLM 调用** | **约 ¥1.5-3.0** |

#### 8.7.2 熔断器如何保护 API Key

熔断器不仅是"防止系统崩溃"，也是对 API Key 的**财务保护**：

```
没有熔断器:
  错误 API Key → 每次调用失败 → 系统继续重试 → 
  线程池被错误请求占满 → 正常请求也无法处理 → 
  更糟的是: 如果是限流(429)而非错误 Key →
  持续重试可能被 API 提供商封禁 Key

有熔断器:
  错误 API Key → 5 次连续失败 → OPEN (15s 静默) →
  所有新消息直接返回 null → 0 次 API 调用 →
  API Key 不会被无效调用浪费 →
  15s 后探测 1 次 → 成功则恢复, 失败则继续静默
```

#### 8.7.3 Semaphore 信号量对 API Key 的保护

```java
// Constants.java
public static final int BOT_MAX_CONCURRENCY = 1;
```

每个 Bot 同时只能有 1 个 LLM 调用进行中。这意味着：

| 场景 | 无 Semaphore | 有 Semaphore(=1) |
|------|------------|-----------------|
| 用户对 Bot 连续发 10 条消息 | 10 次并发 LLM 调用 → 10× API 消耗 | 1 次调用 + 9 条入队 → **前一条回复可能让后一条变缓存命中** |
| 多个用户同时 @Bot | 每个用户触发 1 次调用，可能并发 | 串行处理 → 可能上一个回复就是下一个问题的缓存 |

> Semaphore=1 不仅保证上下文有序性，也间接减少了重复 LLM 调用——因为串行处理时，前一个回复可能让后一个相同问题命中缓存。

### 8.8 记忆系统的一致性保障

#### 并发写入保护

```java
// ConversationMemoryCache.java
// 工作记忆的 Read-Modify-Write 用 per-key mutex 保护
ConcurrentHashMap<String, Object> mutexes = new ConcurrentHashMap<>();

// LongTermMemoryService.java
// 缓存重建防惊群: Redis SETNX stampede lock, 5s TTL
stringRedisTemplate.opsForValue().setIfAbsent("ltm:stampede:" + key, "1", 5, TimeUnit.SECONDS);
```

#### 分布式场景下的记忆一致性

| 组件 | 单机（Simple 模式） | 多机（Relay 模式） |
|------|-------------------|-------------------|
| L0 工作记忆 | Redis String（跨实例共享）✅ | Redis String ✅ |
| L1 短期记忆 | Redis List（跨实例共享）✅ | Redis List ✅ |
| L2 长期记忆 | MySQL + Redis 缓存（SETNX 防惊群） | MySQL + Redis ✅ |
| L3 RAG | MySQL embeddings（单写多读） | 需要额外的向量索引同步策略 ⚠️ |
| 响应缓存 | 本地 LRU（不跨实例）⚠️ | 可考虑 Redis 集中缓存 |
| Skill 缓存 | 本地 LRU + 60s TTL（最终一致） | 变更后需通知其他实例失效 ⚠️ |

> **当前限制**：响应缓存和 Skill 缓存是本地内存缓存，多机部署时存在一致性问题。解决方向：① Redis 集中缓存（增加网络延迟但消除不一致）② 消息总线通知失效（如 Redis Pub/Sub `skill:updated:{id}`）
