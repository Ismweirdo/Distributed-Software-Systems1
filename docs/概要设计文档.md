## 1. 系统概述

### 1.1 系统定位

Chatroom 是一个**分布式即时通讯系统**，在传统 IM 之上集成 **AI Bot 化身系统**。用户可导入聊天记录自动生成具有特定语言风格的 AI 机器人，支持多级记忆（L0-L3）和流式对话。系统采用前后端分离架构，支持 Docker 容器化部署，面向 Redis / MySQL / RabbitMQ 的分布式演进。

### 1.2 技术栈

| 层 | 技术 | 版本 |
|------|------|------|
| 后端框架 | Spring Boot | 3.2 |
| ORM | MyBatis-Plus | — |
| 实时通信 | STOMP over WebSocket | — |
| 数据库 | MySQL (InnoDB) | 8.0 |
| 缓存 | Redis | 7.x (Docker) |
| 消息中间件 | RabbitMQ（可选） | management |
| 前端框架 | Vue 3 + Element Plus + Pinia | — |
| 前端通信 | SockJS + STOMP.js | — |
| LLM SDK | OpenAI 兼容 API（Apache HttpClient 5） | — |
| 构建工具 | Maven（后端）+ Vite（前端） | — |
| 容器化 | Docker Compose | — |
| 语言 | Java 21, Python 3.10+ | — |

### 2.3 WebSocket 双模式

| 模式 | Broker | 路由方式 | 适用场景 |
|------|--------|---------|---------|
| `simple` | 内置内存 STOMP Broker | 单机直接路由 | 开发 / 单机部署 |
| `relay` | RabbitMQ STOMP Relay | 经 RabbitMQ 跨实例路由 | 生产级分布式部署 |

切换方式：环境变量 `CHATROOM_WEBSOCKET_MODE`（simple / relay），无需改代码。

Relay 模式消息流转：

```
Client A ──WS──▶ Server 1 ──STOMP──▶ RabbitMQ ──STOMP──▶ Server 2 ──WS──▶ Client B
```

### 3.2 Bot 生命周期

```
注册 → 配置 Skill → 绑定厂商/Key → 上线运行 ↔ 熔断保护
  │                    │
  ├── 聊天记录导入 (QQ/微信/JSON) → 特征提取 → 自动生成 Bot
  ├── GitHub Skill 导入 / MD 文件导入 → 补充设定
  └── 基准测试 → 延迟/并发评估
```

### 3.3 Bot 消息处理流程

```
用户消息 → STOMP /app/chat.send → 持久化 + WS 推送
  │
  └── 目标是 Bot? → botTaskExecutor 异步
       ├── 熔断检查 (5 次失败 → 15s 静默 → 半开探测)
       ├── 信号量获取 (每 Bot 1 并发)
       ├── buildContext() 构建 LLM 上下文 (7 层组装)
       ├── LLM API 调用 (流式 SSE over WebSocket)
       └── WS 推送回复 → 异步记忆缓存
```

### 3.4 LLM 上下文组装 (buildContext)

```
1. System Prompt (5 层组合):
     ├── Skill 文件夹 (SKILL.md + examples/ + references/ + custom/)
     ├── 情感设定指令 (emotion_profile → 自然语言)
     ├── 语言风格指令 (language_style → 自然语言)
     ├── 对话模式指令 (casual/roleplay/assistant)
     └── compressPrompt() (>8000 字符时保留头 5000 + 尾 3000)

2. Working Memory (L0) — 最近 5 轮完整对话，Redis TTL 30min

3. Long-Term Memory (L2) — 跨会话用户画像，重要性 Top 5 注入

4. Few-Shot Examples — 交替 user/assistant 消息

5. Short-Term Memory (L1) — 最近 30 条消息，Redis TTL 60min

6. RAG Retrieval (L3) — 向量相似度/关键词检索，去重后注入

7. Current Message — 用户当前输入
```

## 4. 多级记忆系统设计

### 4.1 架构总览

模拟人类记忆模型：工作记忆 → 短期记忆 → 长期记忆 → 语义记忆。

```
┌─────────────────────────────────────────────────────────────┐
│                    LLM Context Assembly                     │
│  System Prompt → Working Memory → Long-Term Memory →        │
│  Few-Shot Examples → Short-Term Memory → RAG → Current Msg  │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                     Storage Backends                        │
│  L0 (Working)   Redis String    TTL: 30min  容量: 5 轮      │
│  L1 (Short-Term) Redis List     TTL: 60min  容量: 30 条     │
│  L2 (Long-Term)  MySQL + Redis  TTL: 120min 容量: 50 条     │
│  L3 (Semantic)   MySQL          持久化      容量: 500 条    │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 各层详解

| 层级 | 名称 | 存储 | 容量 | TTL | 注入优先级 | 说明 |
|------|------|------|------|-----|-----------|------|
| L0 | 工作记忆 | Redis String | 5 轮 | 30min | 最高，始终注入 | 最近完整对话，等同于"正在想的事" |
| L1 | 短期记忆 | Redis List | 30 条 | 60min | 中 | 近期消息历史，DB 回退 |
| L2 | 长期记忆 | MySQL + Redis | 50 条/对 | 120min 缓存 | 重要性 Top 5 注入 | fact/preference/summary 三种类型 |
| L3 | 语义记忆 | MySQL | 500 条/对 | 持久化 | Top-K 检索结果注入 | 向量余弦相似度 + 关键词 Jaccard 回退 |

### 4.3 L2 长期记忆类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `fact` | 用户事实信息 | "用户名叫张三，在北京工作，职业是程序员" |
| `preference` | 用户偏好习惯 | "用户喜欢简洁直接的回复，不喜欢客套话" |
| `summary` | 对话主题摘要 | "上次聊了关于 AI 发展趋势的话题" |

### 4.4 记忆整合流程

```
短期记忆 > 20 条
     ↓
maybeConsolidateMemory()
     ↓
memoryCache.trimMemory() — 裁剪最旧 50%
     ↓
LLM 分析对话 → 提取 JSON:
  {facts: [...], preferences: [...], summary: "..."}
     ↓
逐条 storeMemory() → DB + Redis 缓存
     ↓
超过 50 条时淘汰最旧记录
```

### 4.5 降级策略

| 场景 | 行为 |
|------|------|
| Redis 不可用 | 工作记忆跳过；短期记忆回退 MySQL `messages` 表；LTM 直接查 DB |
| LLM 整合失败 | 日志记录警告，不阻塞消息处理；下次对话时重新触发 |
| Embedding API 失败 | RAG 回退关键词 Jaccard 检索（阈值 0.05） |
| DB 写入失败 | 异常向上传播，API 返回 500 错误 |

## 7. 关键设计模式与机制

### 7.1 熔断器状态机

```
CLOSED ──(5 次连续错误)──▶ OPEN ──(15s 静默)──▶ HALF-OPEN
  ▲                         │                      │
  │                         │              (探测成功)──▶ CLOSED
  └─────────────────────────┘              (探测失败)──▶ OPEN
```

- 触发阈值：5 次连续错误
- 静默期：15 秒
- 状态持久化：Redis + DB 双重
- 探测机制：HALF-OPEN 状态下允许 1 条消息穿透

### 7.2 并发控制体系

```
                        ┌─────────────────────────────┐
                        │      STOMP Channel 层        │
                        │  Inbound: core=20, max=100   │
                        │  Outbound: core=20, max=100  │
                        └─────────────┬───────────────┘
                                      │
                        ┌─────────────▼───────────────┐
                        │      信号量并发控制层         │
                        │  per-bot Semaphore(1)        │
                        │  per-group Semaphore(1)      │
                        │  ConcurrentHashMap           │
                        └─────────────┬───────────────┘
                                      │
              ┌───────────────────────┼───────────────────────┐
              │                       │                       │
    ┌─────────▼─────────┐  ┌─────────▼─────────┐  ┌─────────▼─────────┐
    │  Redis 消息队列    │  │  Bot Async 线程池  │  │  HTTP 连接池       │
    │  FIFO List         │  │  core=50, max=300  │  │  maxTotal=300      │
    │  SETNX 分布式锁    │  │  queue=2000        │  │  perRoute=300      │
    │  阻塞出队 BLPOP    │  │  CallerRunsPolicy  │  │  socketTO=180s     │
    └─────────┬─────────┘  └─────────┬─────────┘  └─────────┬─────────┘
              │                       │                       │
    ┌─────────▼───────────────────────▼───────────────────────▼─────────┐
    │                          缓存层                                    │
    │  Skill Cache (500 / TTL 60s)   Response Cache (2000 / TTL 180s)   │
    │  Conversation Memory (Redis / TTL 30-60min)   LTM (MySQL + Redis) │
    └───────────────────────────────────────────────────────────────────┘
                                      │
                        ┌─────────────▼───────────────┐
                        │        降级 / 容错层          │
                        │  熔断器 (5 错 → 15s OPEN)     │
                        │  Redis → 内存队列 自动回退    │
                        │  Redis → DB LTM 自动回退      │
                        └─────────────────────────────┘
```

| STOMP Outbound | 20 | 100 | — | — | 消息推送 |
| taskScheduler | 8 | — | — | — | @Scheduled 定时任务 |

### 7.4 HTTP 连接池配置

| 参数 | 值 | 设计考量 |
|------|-----|---------|
| maxTotal | 300 | 200+ Bot 同时在线需要至少 200 连接，留 50% 余量 |
| defaultMaxPerRoute | 300 | 所有 Bot 通常调用同一 LLM API 端点 |
| connectTimeout | 30s | TCP 握手超时 |
| socketTimeout | 180s | LLM 流式生成可能持续几十秒到几分钟 |

### 7.5 缓存体系

| 缓存 | 容量 | TTL | 存储 | 作用 |
|------|------|-----|------|------|
| Skill 缓存 | 500 条 | 60s | 内存 LRU | 避免每次查 BotSkill 走 DB |
| 响应缓存 | 2000 条 | 180s | 内存 LRU | 相同问题返回缓存，避免重复 LLM 调用 |
| 短期记忆 | 30 条 | 60min | Redis List | 最近 N 轮对话上下文 |
| 长期记忆 | 50 条/对 | 120min 缓存 | MySQL + Redis | 固化的重要记忆 |
| 工作记忆 | 5 轮 | 30min | Redis String | 当前会话活跃上下文 |

## 10. 分布式演化路线

系统设计为逐步向分布式演进：

- **阶段 1（当前）**：单机部署，Simple STOMP Broker，Redis 消息队列，MySQL 单实例
- **阶段 2**：引入 RabbitMQ Relay，多 Server 实例水平扩展
- **阶段 3**：MySQL 读写分离 / 分库分表；Redis Cluster
- **阶段 4**：Kubernetes 编排，全链路监控（Prometheus + Grafana）
