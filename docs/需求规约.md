## 1. 引言

### 1.1 项目背景

本项目是一个**分布式即时通讯系统**，核心特色是在传统 IM 功能之上集成 **AI Bot 化身系统**——用户可导入聊天记录自动生成具有特定人物语言风格的 AI 机器人，支持多级记忆（工作记忆/短期记忆/长期记忆/RAG 语义检索）和流式对话。

### 1.2 项目范围

系统提供完整的即时通讯能力（用户管理、好友、群组、实时消息），并在此基础上构建 AI Bot 共存生态：Bot 注册与配置、技能蒸馏与导入、多级记忆系统、多厂商 LLM 切换、流式对话、熔断保护、主动聊天等。

### 1.3 术语定义

| 术语 | 说明 |
|------|------|
| Bot | AI 机器人，具有独立人格和语言风格的虚拟用户 |
| Skill | Bot 的技能配置，含系统提示词、情感设定、语言风格等 |
| 蒸馏 (Distill) | 从聊天记录中自动提取语言风格和人格特征 |
| LTM | Long-Term Memory，长期记忆 |
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| STOMP | Streaming Text Oriented Messaging Protocol，基于 WebSocket 的消息协议 |
| SSE | Server-Sent Events，服务端推送事件 |

| FR-USER-05 | 用户可修改个人资料（头像、昵称、密码） | 中 |

| FR-FRIEND-02 | 接收方可接受或拒绝好友请求 | 高 |
| FR-FRIEND-03 | 用户可删除好友 | 中 |
| FR-FRIEND-04 | 用户可为好友（含 Bot）设置备注名 | 中 |
| FR-FRIEND-05 | 好友关系为双向唯一约束 | 高 |

### 2.3 群组管理 (FR-GROUP)

| 编号 | 需求 | 优先级 |
|------|------|--------|

| FR-GROUP-02 | 群主可踢出成员、转让群主、解散群组 | 中 |
| FR-GROUP-03 | 群成员角色分为 owner/admin/member | 中 |
| FR-GROUP-04 | Bot 可加入群组并自动回复群消息 | 中 |

### 2.4 实时消息 (FR-MESSAGE)

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-MESSAGE-01 | 系统支持基于 WebSocket + STOMP 协议的实时消息收发 | 高 |
| FR-MESSAGE-02 | 支持私聊（点对点）消息路由 | 高 |
| FR-MESSAGE-03 | 支持群聊消息路由 | 高 |

| FR-MESSAGE-07 | 消息需包含已读/未读状态 | 低 |

### 2.5 Bot 管理 (FR-BOT)

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-BOT-01 | 用户可注册 Bot，配置其基础信息 | 高 |
| FR-BOT-02 | 支持通过聊天记录导入（QQ/微信/JSON）自动生成 Bot | 高 |
| FR-BOT-03 | 支持从 GitHub URL 或 MD 文件导入 Skill 配置 | 高 |
| FR-BOT-04 | Skill 导入默认合并模式，可切换为覆盖模式 | 中 |
| FR-BOT-05 | Bot 支持配置独立的 LLM 厂商、API Key、模型 | 高 |
| FR-BOT-06 | Bot 支持配置对话模式：casual（轻松闲聊）、roleplay（沉浸扮演）、assistant（专业助手） | 高 |
| FR-BOT-07 | Bot 支持上传自定义头像 | 低 |
| FR-BOT-08 | 系统可查看 Bot 列表、活跃 Bot 列表、Bot 数量统计 | 中 |
| FR-BOT-09 | Bot 支持永久删除 | 中 |

### 2.6 AI 厂商与模型 (FR-PROVIDER)

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-PROVIDER-01 | 系统预置 7 家 AI 厂商配置：DeepSeek、Kimi、Qwen、Mimo、OpenAI GPT、智谱GLM、自定义 | 高 |
| FR-PROVIDER-02 | 每家厂商支持多个模型选项，用户可切换 | 高 |
| FR-PROVIDER-03 | Bot 运行时支持切换厂商/Key/模型 | 中 |

### 2.7 Skill 体系 (FR-SKILL)

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-SKILL-01 | Skill 支持 11 维数据分区：system_prompt、emotion_profile、language_style、tone_signature、rhythm_profile、discourse_tactics、topic_preferences、safety_boundaries、repair_strategy、example_guidelines、few_shot_examples | 高 |
| FR-SKILL-02 | 聊天记录导入时自动提取情绪特征（六维正则匹配）和语言风格统计 | 高 |
| FR-SKILL-03 | Skill 采用双存储模型：数据库 `bot_skills` 表（运行时） + `data/skills/{name}/` 文件夹（源文件） | 中 |
| FR-SKILL-04 | Skill 支持预热加载（@PostConstruct），消除首次请求冷启动延迟 | 中 |
| FR-SKILL-05 | Skill 缓存 TTL 60s，容量 500 条，LRU 淘汰 | 中 |

### 2.8 多级记忆系统 (FR-MEMORY)

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-MEMORY-01 | **L0 工作记忆**：最近 5 轮完整对话，Redis 存储，TTL 30min，始终注入 LLM 上下文 | 高 |
| FR-MEMORY-02 | **L1 短期记忆**：最近 30 条消息，Redis List 存储，TTL 60min，DB 回退 | 高 |
| FR-MEMORY-03 | **L2 长期记忆**：跨会话用户画像，MySQL + Redis 缓存，每对 50 条上限，支持 fact/preference/summary 三种类型，重要性 1-5 评分 | 高 |
| FR-MEMORY-04 | **L3 语义记忆(RAG)**：向量相似度检索，MySQL embeddings 存储，每对 500 条上限，支持关键词 Jaccard 回退 | 高 |
| FR-MEMORY-05 | 短期记忆超过 20 条时自动触发 LLM 记忆整合（Consolidation） | 高 |
| FR-MEMORY-06 | L2 长期记忆支持手动触发整合 | 中 |
| FR-MEMORY-07 | 所有记忆层级均具备 Redis → MySQL 降级策略 | 高 |

### 2.9 LLM 调用与流式响应 (FR-LLM)

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-LLM-01 | LLM 上下文构建按 7 层顺序组装：System Prompt → Working Memory → Long-Term Memory → Few-Shot Examples → Short-Term Memory → RAG Retrieval → Current Message | 高 |
| FR-LLM-02 | 支持基于 SSE 的流式响应，通过 WebSocket 推送给客户端 | 高 |
| FR-LLM-03 | 超长 System Prompt（>8000 字符）自动压缩：保留头部 5000 + 尾部 3000 | 中 |
| FR-LLM-04 | 响应缓存：hash(botUserId+content) 为 key，TTL 180s，容量 2000，LRU 淘汰 | 高 |

### 2.10 Bot 并发控制与容错 (FR-RELIABILITY)

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-RELIABILITY-01 | 每 Bot 独立信号量（Semaphore=1），保证消息串行处理、上下文有序 | 高 |
| FR-RELIABILITY-02 | Redis 消息队列（FIFO List），每 Bot 最大 200 条，阻塞出队（BLPOP 语义） | 高 |
| FR-RELIABILITY-03 | 熔断器：5 次连续错误 → OPEN（15s 静默）→ HALF-OPEN → 探测成功 → CLOSED，状态持久化至 Redis + DB | 高 |
| FR-RELIABILITY-04 | 分布式锁：Redis SETNX 保证多实例场景下同一 Bot 队列不被并发处理 | 中 |
| FR-RELIABILITY-05 | Redis 不可用时自动降级：消息队列退化为 ConcurrentLinkedQueue 内存队列，记忆系统回退 DB | 高 |
| FR-RELIABILITY-06 | 异步非阻塞写入：消息持久化、RAG Embedding 存储、LTM 记忆固化均通过 CompletableFuture 异步执行 | 中 |

### 2.11 主动聊天 (FR-ACTIVE)

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-ACTIVE-01 | Bot 可定时向随机好友发起话题，间隔 15s~600s 可配置 | 中 |
| FR-ACTIVE-02 | 熔断 Bot 自动跳过 | 中 |
| FR-ACTIVE-03 | 支持群聊自动回复 | 低 |

### 2.12 监控与基准测试 (FR-MONITOR)

| 编号 | 需求 | 优先级 |
|------|------|--------|
| FR-MONITOR-01 | 提供系统健康检查端点（DB 状态、Redis 状态） | 高 |
| FR-MONITOR-02 | 提供消息队列统计端点 | 中 |
| FR-MONITOR-03 | 提供 Bot 基准测试服务：p50/p90/p99 延迟指标 | 中 |
| FR-MONITOR-04 | 提供快速延迟检查端点 | 低 |

### 3.1 性能需求 (NFR-PERF)

| 编号 | 指标 | 目标值 | 实测值 |
|------|------|--------|--------|
| NFR-PERF-01 | REST API QPS | ≥ 500 | 1226 |
| NFR-PERF-02 | REST API p95 延迟 | ≤ 100ms | 16ms |
| NFR-PERF-03 | WebSocket 最大并发连接 | ≥ 500 | 1000 |
| NFR-PERF-04 | 消息吞吐 | ≥ 500 msg/s | 951 msg/s |
| NFR-PERF-05 | Bot 回复率（20 Bot × 3 轮） | 100% | 100% |
| NFR-PERF-06 | 稳定并发上限（0% 错误率） | ≥ 100 | 290 |

### 3.2 可用性需求 (NFR-AVAIL)

| 编号 | 需求 | 说明 |
|------|------|------|
| NFR-AVAIL-01 | Redis 不可用时系统可降级运行 | 消息队列退化为内存队列；记忆系统回退 DB |
| NFR-AVAIL-02 | 单个 Bot 故障不影响其他 Bot | 熔断器隔离 + 错误隔离 |
| NFR-AVAIL-03 | LLM API 不可用时 Bot 自动熔断 | 5 次连续错误触发，15s 静默后探测恢复 |
| NFR-AVAIL-04 | 登录保护：IP 限流 + 失败计数 + 账号锁定 | Caffeine/Redis 双层降级 |

### 3.3 安全需求 (NFR-SEC)

| 编号 | 需求 | 说明 |
|------|------|------|
| NFR-SEC-01 | 所有 API 需 JWT Token 认证 | 无状态 Session |
| NFR-SEC-02 | 密码使用 BCrypt 加密存储 | — |
| NFR-SEC-03 | WebSocket 连接需 JWT 握手校验 | AuthHandshakeInterceptor |
| NFR-SEC-04 | API Key 敏感信息加密存储 | 环境变量或前端独立配置 |
| NFR-SEC-05 | 文件上传需路径遍历防护 + 类型校验 | — |

| NFR-SCALE-03 | 支持多厂商 LLM 热切换 | 7 家预置厂商 + 自定义 |

## 4. 系统用例

### 4.1 普通用户用例

```
用户 ──┬── 注册/登录
       ├── 管理好友（添加/接受/拒绝/删除/备注）
       ├── 管理群组（创建/解散/踢人/转让）
       ├── 实时聊天（私聊/群聊，文本/图片/文件）
       ├── 注册 & 配置 AI Bot
       ├── 导入聊天记录生成 Bot
       ├── 导入 Skill 文件定制 Bot 人格
       ├── 切换 Bot 的 AI 厂商/模型
       ├── 查看 Bot 长期记忆
       └── 触发 Bot 基准测试
```

### 4.2 Bot 用例

```
Bot ──┬── 接收用户消息 → LLM 生成回复 → 推送
       ├── 多级记忆：工作记忆 ↔ 短期记忆 ↔ 长期记忆
       ├── RAG 语义检索增强回复
       ├── 熔断自保护（故障时自动静默）
       ├── 主动发起聊天（定时向随机好友发话题）
       └── 群聊自动回复
```

## 5. 数据流概要

### 5.1 消息处理主流程

```
用户消息 → STOMP /app/chat.send
  ├── 持久化到 MySQL
  ├── WebSocket 推送给目标用户
  └── 目标是 Bot？
       └── botTaskExecutor 异步处理
            ├── 熔断检查
            ├── 信号量获取 (per-bot Semaphore=1)
            ├── buildContext() 组装 LLM 上下文 (7层)
            ├── LLM API 调用 (流式 SSE over WebSocket)
            └── WS 推送回复 → 异步记忆缓存
```

### 5.2 记忆整合流程

```
短期记忆 > 20 条
  → maybeConsolidateMemory()
  → memoryCache.trimMemory() (裁剪最旧 50%)
  → LLM 分析对话 → 提取 JSON: {facts, preferences, summary}
  → 逐条 storeMemory() → DB + Redis 缓存
  → 超过 50 条时淘汰最旧记录
```
